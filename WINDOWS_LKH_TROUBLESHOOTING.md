# Windows 专用排查清单（LKH 崩溃 vs 数据参数崩溃）

## 1) 最小可复现

- 固定同一条实例（同一输入、同一 `.par/.vrp`），单次运行，不并发。
- 记录以下信息（代码已自动打印）：
  - `command`
  - `returncode`（含 16 进制）
  - `stdout/stderr` 首尾 50 行
  - 临时文件路径：`par_path` / `vrp_path` / `tour_path`

## 2) 确认可执行文件本体

- 用绝对路径确认 Python 侧实际调用的可执行文件（日志里的 `lkh_path`）。
- 在 Windows 终端手工运行：`LKH-3.exe test.par`（不经过 Python）。
- 若手工也崩溃，优先判定为 **LKH/运行环境问题**。

## 3) 返回码快速判定（Windows 关键）

- `-1073741819` (`0xC0000005`)：访问冲突，典型 **进程崩溃**。
- `-1073740791` (`0xC0000409`)：栈/缓冲区异常，偏 **进程崩溃**。
- 普通非零退出且有参数类报错：更可能 **数据/参数问题**。

## 4) 区分“二进制崩”还是“数据触发崩”

- 同一个 exe + 不同输入：
  - 所有输入都崩：**二进制/运行库/权限/路径问题**。
  - 仅个别输入崩：**数据参数触发崩溃**（边界条件）。
- 同一个输入 + 不同机器：
  - 仅某台机崩：**环境问题**（VC 运行库、杀软、路径权限）。

## 5) 数据参数专项检查（最常见）

- `DIMENSION`、节点编号、坐标、`DEMAND`、`CAPACITY` 是否一致且无越界。
- `VEHICLES` 是否过小导致不可行；压缩后节点/需求是否仍合法。
- `.par` 引用路径是否正确（Windows 下空格/中文路径优先改英文短路径验证）。

## 6) 路径与权限检查（Windows 特有）

- 项目目录、临时目录避免中文/空格/超长路径（先用短英文路径复现）。
- 以普通用户和管理员各试一次。
- 关闭实时杀软扫描后重试（仅用于排查）。

## 7) 并发与临时文件冲突检查

- 禁用并发，确认临时 `.par/.vrp/.tour` 文件名唯一。
- 若单线程稳定、多线程不稳：判定为 **并发文件冲突/竞争**。

## 8) 最终判定规则（可直接用）

- 手工运行也崩 + Windows 异常码（`0xC0000005` / `0xC0000409`） => **LKH 本体/环境崩溃**。
- 仅特定实例触发，且调整 `VEHICLES/CAPACITY` 或输入后恢复 => **数据参数崩溃**。
- 单线程稳定、多线程崩 => **并发临时文件问题**。

## 9) 当前仓库已落地的保护措施

- 非零退出会自动分类并打印诊断日志。
- 遇到疑似数据/参数错误会自动进行 1 次降级重试（放宽 `VEHICLES`）。
- 仍失败时自动回退到 fallback 方案，避免整批任务中断。
